直接从实际开发场景出发，200K token 大概等于 **15万汉字 / 约10万行代码**，这个量级在工程实践中意味着什么：

| 场景 | 单个复杂文件 | 中型服务 | 大型单体项目 |
|------|------------|--------|------------|
| 文件数 | 1 个 | 30–80 个 | 200–500 个 |
| token 量 | ≈ 5K | ≈ 60–100K | 300K–1M |
| 能否放入 | 轻松放入 | 完整放入 | 需要切分 |
| 典型任务 | 重构 / 加注释 | 跨文件 bug / 重构 | 需配合 RAG / 索引 |

> 200K ≈ 15 万汉字 ≈ 约 8–10 万行代码

上图说明了容量边界。接下来是最核心的问题：**上下文大，到底让哪些以前做不到的事情变得可能？**

---

**1. 跨文件 bug 追踪**

以前模型只能看单个文件，遇到"A 调用 B，B 调用 C，C 的类型定义在 D 里"这种依赖链，它根本看不完整，只能给你一个在局部看起来合理但实际不对的答案。200K 上下文可以把整条调用链相关的文件一次性放进去，模型才能真正理解"数据从哪来、在哪变形、在哪出了问题"。

**2. 全局重构不遗漏**

把一个函数改名、把一个接口签名改变，影响面可能散落在几十个文件里。以前只能靠 IDE 的符号引用做机械替换，复杂的语义变化（比如改了一个 enum 的含义，需要同步改所有 switch 里的逻辑）就抓瞎了。有了完整上下文，模型能告诉你"这 12 处需要改，其中第 5 处语义上还需要额外处理"。

**3. 补写测试时理解真实逻辑**

为已有代码补测试，最难的不是写测试语法，而是理解"这个函数在什么边界条件下会有特殊行为"。这要求模型看到整个函数的上下游，以及它依赖的数据结构定义。以前只能给个模板，现在能给出真正覆盖边界的 test case。

**4. 文档生成不靠猜**

生成 API 文档、写 README，过去模型只能看函数签名猜用途。把整个模块上下文放进去，它能理解这个函数在整个业务流程中的位置，写出来的文档才有意义，而不是"这个函数接受一个参数并返回一个值"这种废话。

**5. 代码审查有全局视角**

PR review 时，如果只给模型看 diff，它无法判断"这个改动是否破坏了其他地方的假设"。把相关模块一起放进去，它才能做真正有价值的审查，而不只是检查语法和命名。

---

**但也要知道它的真实局限：**

200K 是容量上限，不是质量保证。模型对放在上下文**开头和结尾**的内容注意力更集中，中间部分的长段代码有时候会被"稀释"。这个现象叫 **lost in the middle**，在实践中意味着：如果你把最关键的代码扔在一大段内容的中间，效果会比预期差。应对方法是把核心文件放在输入的前面或后面，相关背景放中间。

另外，超过 200K 的大型单体项目（比如 Linux kernel、大型商业系统）还是需要配合 RAG（检索增强）或代码索引工具，先检索出相关片段再喂给模型，而不是无脑把整个库塞进去。
